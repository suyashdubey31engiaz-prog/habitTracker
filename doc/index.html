Habitual

A full-featured personal wellness and habit tracking app. Build daily habits, visualise streaks and progress with rich charts, get AI-powered coaching that knows your health profile, and watch the UI adapt to your local weather and season in real time — complete with canvas-rendered weather effects and 17 switchable themes.

🛤️Build Journey

This project started as a simple habit checkbox list and grew iteratively into a full wellness platform. Each phase added a major new capability.

Phase 1 — Core MERN Foundation

JWT auth, MongoDB Atlas, habit CRUD, daily check-in with optimistic updates, progress ring, streaks. Basic Layout with bottom nav.

Phase 2 — Views & Analytics

Weekly bullet journal grid, full monthly checkbox grid, Analytics page with heatmap, bar, line, pie, and radar charts. 28-day data window with Day/Week/Month/Year tabs.

Phase 3 — Themes, Fonts & Polish

17 CSS-variable themes, 10 Google Font pairings, all saved in localStorage. Card variants per theme (neon glow, aurora animated gradient border, cyberpunk clip-path). Page transition animations on route change.

Phase 4 — Weather & Seasons

OpenWeatherMap integration, GPS geolocation with IP fallback, canvas weather effects (rain, snow, fog, storm lightning, stars, heat shimmer, frost crystals). Season auto-detection, season particle effects per theme.

Phase 5 — AI Coach & User Profile

Groq API (free Llama 3 model) powering a context-aware chatbot. 6-step About wizard collecting age, diet, occupation, fitness level, health goals, allergies, BMI. AI uses all profile data + current habits for personalized advice.

Phase 6 — PWA & Offline Sync

Service worker, web app manifest, installable on Android and iOS. IndexedDB offline queue — writes are stored on device and auto-replayed when network returns. Sync banner with manual sync button.

Phase 7 — Deploy & Fix

Deployed backend on Render, frontend on Vercel. Fixed production CORS issue (CLIENT_URL mismatch), fixed service worker intercepting cross-origin API calls returning 404, added vercel.json for SPA routing.

⚠️ Timezone Safety

Dates are stored as local-time strings (YYYY-MM-DD) using a getLocalDateStr() helper, never as UTC ISO strings. This prevents the common timezone drift bug where checking in at 11 PM logs to the wrong calendar day.

🌤️Weather System

1
Location Detection

WeatherContext tries navigator.geolocation.getCurrentPosition() first. On permission denial, falls back to ipapi.co/json/ for city name, then calls OpenWeatherMap by city. Results cached in localStorage for 30 minutes to avoid hammering the free tier limit (1000 calls/day).

2
Effect Mapping

OWM condition codes are mapped to 8 internal effect names: sunny | rain | snow | fog | storm | night | cloudy | clear. Night is triggered when local hour < 6 or ≥ 20. Each effect drives both a canvas/CSS overlay and a set of habit recommendations.

3
Canvas Effects

Rain: 80 animated line particles on a full-screen canvas. Snow: 60 circular flakes with drift. Storm: heavy rain (140 particles) + lightning flash overlay that fires randomly every 3–8 seconds with double-flash simulation. Night: 40 twinkling star dots positioned in the top 40% of the screen.

4
UI Integration

Temperature chip in the header shows live °C with a comfort-coded colour (blue for cold, green for pleasant, orange for warm, red for hot). Dashboard shows a weather recommendation card with habit suggestions relevant to current conditions. Weather + season context is also passed to the Groq AI for smarter advice.

🤖AI Coach

Powered by Groq API (free tier — Llama 3.3 70B model, 6000 tokens/min, 30 req/min). The AI is not generic — it has full context of the user's life built into every system prompt.

1
Dynamic System Prompt

On every chat request, buildSystemPrompt(userId) fetches the user's full profile from MongoDB plus all active habits and their 7-day completion stats. This context is injected into the system prompt — so when you ask "what should I eat today?", the AI knows you're vegetarian, 21 years old, a student in Lucknow, trying to lose weight, and that you only completed 3/7 habits last week.

2
Three Endpoints

/chat — full conversational AI, last 20 messages sent for context window management. /tips — returns 3 personalized JSON tips on page load (parsed from Groq JSON response). /food — suggests 4 meals with name, emoji, calorie estimate, prep time, and reason — all respecting dietary restrictions.

3
Voice Input & Output

Voice input uses the Web Speech API (webkitSpeechRecognition) with lang=en-IN. Interim results update the input field in real time. On final result, the message is auto-sent. Text-to-speech uses SpeechSynthesisUtterance — clicking the speaker icon reads the last AI reply aloud with natural Indian English intonation.

4
Two-Column Layout

Desktop: sidebar (1fr) on the left shows daily tips as clickable cards and 8 quick-prompt buttons. Chat window (2.2fr) on the right fills the remaining width with the message area at full viewport height. On mobile (<680px) it stacks vertically with chat on top.

💡 Streak Algorithm

Streaks are computed server-side in routes/logs.js from the full 28-day window. The algorithm groups logs by date, checks consecutive completion, and also computes per-habit streaks and best-ever streak — all in a single MongoDB query + in-memory pass to avoid N+1 queries.

📱PWA & Offline Sync

📱 Progressive Web App

  • manifest.json — name, short_name, theme_color, display:standalone
  • Service worker registered on window.load
  • Android: Chrome shows native install banner (beforeinstallprompt event)
  • iOS: InstallPrompt detects Safari + not standalone → shows Share → Add to Home Screen instructions

🔄 Offline Sync

  • Axios response interceptor catches network errors on mutations (POST/PUT/DELETE)
  • Failed writes are stored in IndexedDB (habitual_offline database)
  • window.addEventListener('online') triggers automatic replay
  • SyncBanner polls queue size every 5 seconds
  • Manual "Sync Now" button in Profile for edge cases
  • Fake success returned to caller — UI stays consistent while offline
⚠️ Service Worker Cross-Origin Fix

The service worker only intercepts same-origin requests. All cross-origin requests (Render backend, OpenWeatherMap, ipapi.co) are passed through with a direct fetch(e.request) call. This was the root cause of the 404 production login failure — the old SW was intercepting Render API calls and returning cached 404 responses.

⚡Challenges Faced

🕐
Timezone Date Drift

Using new Date().toISOString() returns UTC time. At 11 PM IST, logging a habit would save to the previous day's date in UTC (+5:30 offset).

Replaced all date handling with a getLocalDateStr() helper using getFullYear(), getMonth(), getDate() — local time, not UTC. Applied everywhere: Dashboard, WeeklyView, MonthlyGrid, and backend stats.
🌐
Service Worker 404 in Production

The SW was intercepting all fetch events including calls to onrender.com. It had no cached response → returned 404. Login and register both failed silently on Vercel.

Added if (url.origin !== self.location.origin) { fetch(e.request); return; } — only same-origin requests are handled by the SW now. Cross-origin passes straight through.
🔒
CORS in Production

Backend was configured with origin: CLIENT_URL. When Vercel URL wasn't exactly matching (trailing slash, wrong URL), all requests were blocked with CORS error.

Updated server.js to check origin.startsWith(CLIENT_URL) and added an explicit allowed origins array covering both localhost variants. Set CLIENT_URL precisely in Render env vars.
🔁
Habit Reorder Route Conflict

Express was matching PUT /api/habits/reorder against the PUT /api/habits/:id route — treating the string "reorder" as a MongoDB ObjectId, causing a CastError.

Moved the /reorder route handler above the /:id handler in habits.js. Express routes match in declaration order — specific paths must always precede parameterised ones.
⚡
Groq Rate Limit Errors

Free tier has a 6000 tokens/minute cap. Long system prompts with full habit context were consuming most of the budget per request, causing 429 errors on rapid successive messages.

Truncated chat history to last 20 messages only. Added hardcoded fallback tips array so the /tips endpoint degrades gracefully. Added error boundary in AIChat.jsx showing a friendly retry message.
🎨
Neon Scanlines Not Global

The neon scanline effect was added to document.body inside Profile.jsx render — it only applied while on the Profile page and disappeared on navigation.

Moved all body class management (neon-scanlines, cyberpunk-glitch, theme-winter, etc.) into the useEffect inside ThemeContext.jsx so it fires on every theme change and applies globally regardless of current page.
📱
Render Free Tier Cold Starts

Render's free tier spins down after 15 minutes of inactivity. First request after sleep takes 30–50 seconds, causing the login to time out on Axios's 8-second timeout setting.

Added a /api/health endpoint. Registered the backend URL on UptimeRobot (free) to ping every 5 minutes — keeps the server warm 24/7 without any paid plan.
🔄
N+1 Query in Stats Endpoint

Early version of /logs/stats ran a separate MongoDB query for each habit to compute its streak, causing 6–10 DB round trips per stats call.

Replaced with a single HabitLog.find() covering the full 28-day window, then grouped results in-memory by habitId using a JavaScript reduce. One DB query total regardless of habit count.

🌐Deployment

Backend → Render (Free)

  • New Web Service → connect GitHub repo
  • Root Directory: backend
  • Build Command: npm install
  • Start Command: node server.js
  • Add all 5 env vars in the Environment tab
  • Copy the deploy URL (e.g. https://habitual-backend-xxxx.onrender.com)
  • Register on UptimeRobot to ping /api/health every 5 min — prevents cold starts

Frontend → Vercel (Free)

  • New Project → import GitHub repo
  • Root Directory: frontend
  • Framework Preset: Vite (auto-detected)
  • Add VITE_API_URL = your Render URL + /api
  • Add VITE_WEATHER_KEY
  • vercel.json already in repo — rewrites all routes to index.html
  • After deploy: update CLIENT_URL on Render to match Vercel URL, then trigger Render redeploy
⚠️ Important Order

Deploy backend first → copy the Render URL → set it in Vercel env vars → deploy frontend → copy the Vercel URL → update CLIENT_URL on Render → Render redeploys. If CLIENT_URL doesn't match exactly, CORS blocks all requests. No trailing slashes.
